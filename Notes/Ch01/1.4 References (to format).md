Ниже — перечень внешних источников, на которые регулярно ссылается JLS. В справочнике даются краткие пояснения, **зачем они нужны разработчику**.

### 1.4.1 Нормативные ссылки (ядро)

- **Unicode Standard** — кодовые точки, классы символов, нормализация. Важен для лексики, идентификаторов, эскейпов.
    
- **IEEE 754 Floating-Point** — форматы и поведение операций с плавающей точкой (`float`, `double`), NaN/Infinity, округление, точность.
    
- **ISO 8601 (даты/время)** — не часть JLS, но косвенно влияет на форматирование и API (java.time).
    
- **The Java Virtual Machine Specification (JVMS)** — правила формата class-файла, байткоды, верификация, модель исполнения.
    

### 1.4.2 Информативные ссылки (ориентиры для практики)

- **Java SE API** (документация модулей/пакетов/классов) — поведение стандартных классов, на которые ссылается JLS (например, `Object`, `String`, `Thread`).
    
- **JDK Tools & Guides** — руководство по инструментам (`javac`, `java`, `jar`, `jlink`, `jpackage`, `jcmd`, `jfr` и др.).
    
- **Release Notes / JEPs** — происхождение и состояние фич (Preview, Incubator, Standard).



### 1.4.3 Дополнительные обязательные источники

- **The Java Virtual Machine Specification (JVMS)** — форматы class-файлов, правила верификации и выполнения байткода.  
  JLS определяет корректность исходного кода, но передача управления на JVM делает JVMS обязательной частью общей спецификации платформы.

- **Java Memory Model (JMM)** — часть JLS (Ch. 17), но ссылается на отдельные формальные публикации о модели памяти.  
  Для корректного понимания многопоточной семантики следует учитывать эти публикации.

---

### 1.4.4 Приложения JLS (Appendices A–Z)

- **Формальные грамматики** — полное описание лексической и синтаксической структуры языка.  
- **Глоссарий терминов** — унифицирует терминологию для всех глав спецификации.  
- **История изменений** — даёт возможность понять эволюцию правил и совместимость версий.  

> При изучении спорных вопросов (например, грамматического разбора) полезно сверяться с формальными грамматиками в приложениях, а не только с текстом глав.

---

### 1.4.5 Ссылки на JEP и Release Notes

- **JEP (JDK Enhancement Proposals)** — источник информации о мотивации, истории и ограничениях фич.  
  Даже после того как фича попадает в JLS, полезно свериться с соответствующим JEP, чтобы понимать решения дизайнеров языка.

- **Release Notes** — фиксируют, в какой версии языка функция активирована как preview, incubator или стандартная.  

> Для работы с современными версиями Java важно проверять статус фичи (Preview / Incubator / Standard) в JEP и Release Notes.

---

### 1.4.6 Практические рекомендации

- При изучении спорного поведения следует сверяться одновременно с JLS и JVMS: первое даёт семантику исходного кода, второе — требования к JVM.  
- Для анализа грамматики и сложных ограничений (например, definite assignment) полезно обращаться к формальным приложениям JLS.  
- Для новых функций обязательно сверяйтесь с JEP, чтобы учесть историю и ограничения, особенно если проект использует preview или incubator-фичи.
